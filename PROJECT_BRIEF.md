# 路书匠项目立项清单（简化版）

## 1. 项目命名

推荐项目名：路书匠

英文名：RouteCraft

一句话定位：

用户输入起点、目的地、范围和交通方式后，系统通过大模型流式生成一份可直接参考的出行规划，内容包含天气预警、路线/公共交通规划、途径景点和实时信息检索。

备选名称：

- 行程铸造局
- 路线制图所
- 出发计划局
- 路书工厂
- 一路成行
- 行迹师
- 途编
- 云路书

建议使用“路书匠”，原因是它强调规划结果，不像复杂 SaaS，也不会显得过度 AI 化。

## 2. 项目目标

### 2.1 产品目标

- 做一个轻量的 AI 出行规划工具。
- 用户不需要复杂下单流程，只需要提交出行条件。
- 系统以流式方式展示大模型规划过程和最终结果。
- 后台只保留用户管理、生成记录管理和 LLM 配置管理。
- 第一版优先验证规划体验、生成速度和结果可用性。

### 2.2 用户输入

必填：

- 起点：城市、地址、景点或站点。
- 目的地：城市、地址、景点、区域或主题描述。
- 范围：可接受的距离、时间范围、游玩半径或天数。
- 交通方式：自驾、公共交通、步行/骑行、混合出行。

选填：

- 出行日期或大致时间。
- 人数。
- 偏好：自然风光、人文、亲子、美食、小众路线、低强度等。
- 避免项：不走高速、少换乘、少步行、避开热门景点等。

### 2.3 系统输出

通过 LLM 流式输出：

- 需求理解和规划假设。
- 天气预警：天气概况、恶劣天气提醒、出行风险。
- 路径点规划：适用于自驾、步行、骑行或混合路线。
- 公共交通规划：适用于地铁、公交、火车、高铁、打车接驳。
- 高德路径图：生成可打开的高德路线链接，并导出路线图预览或截图。
- 途径景点：推荐途经点、停留建议、顺路理由。
- 实时信息检索：分为“新闻/交通管制”和“攻略/避坑参考”两类，补充目的地或沿途的近期资讯、活动、交通管制、游玩攻略和避坑提醒。
- 时间安排：按半日、全天或多日输出行程节奏。
- 风险提示：营业时间、限流、天气、交通、公共交通末班时间等。
- 可复制的结构化结果：Markdown 和 JSON 摘要。

## 3. 核心原则

- 先做简单可用，不做复杂订单、套餐、支付、人工审核。
- 用户端围绕“提交需求、看流式生成、查看历史记录”设计。
- 管理后台只做“用户管理、生成记录管理、LLM 配置管理”。
- 大模型是核心体验，高德、天气、实时信息检索等数据源作为辅助增强。
- 所有外部数据都标注来源和更新时间。
- 第一版不追求完整路书交付，只追求规划建议清晰、可读、可继续追问。

## 4. 技术栈

### 4.1 后端

- Python 3.12+
- FastAPI
- SQLAlchemy 2
- Alembic
- MySQL
- Redis，按需用于缓存和限流
- Pydantic v2
- httpx
- SSE 或 WebSocket，用于流式生成

### 4.2 Web 前端

- Vue 3
- Vite
- Pinia
- Vue Router
- Element Plus 或 Naive UI
- Markdown 渲染组件
- 高德 JS API，按需用于地图展示

### 4.3 第三方能力

- 大模型 API：需求理解、路线规划、解释生成、流式输出。
- 高德 Web 服务 API：地理编码、POI 搜索、路径规划。
- 高德 URI、静态地图或 JS API 截图：生成高德路线链接和路径图导出。
- 天气 API：目的地和沿途天气预警。
- 实时信息检索 API 或搜索服务：目的地近期新闻、交通管制、活动资讯、攻略和避坑参考。

### 4.4 本地服务端口

- 后端 FastAPI 默认端口：`3002`。
- Web 用户端 Vite 默认端口：`3003`。
- 管理后台 Vite 默认端口：`3004`。
- 前端接口请求默认指向 `http://localhost:3002`，后续通过环境变量覆盖。

## 6. 角色设计

只保留两类角色：

- 游客/用户：提交出行规划需求、查看流式生成过程、查看生成结果、管理自己的历史记录。
- 管理员：管理用户、查看生成记录、配置 LLM API 和 Key、查看调用状态、处理异常记录。

本地初始化默认管理员账号：`admin` / `admin123456`。密码入库必须哈希存储，生产环境首次部署后必须修改。

不做：

- 规划师角色。
- 运营角色。
- 复杂 RBAC 权限模型。
- 支付、订单、套餐、退款流程。

## 7. 核心流程

### 7.1 用户生成流程

1. 用户进入规划页。
2. 填写起点、目的地、范围、交通方式。
3. 按需填写日期、人数、偏好和避免项。
4. 点击生成。
5. 后端创建生成记录。
6. 后端聚合外部数据：地图、天气、实时信息。
7. 后端调用 LLM，以 SSE 或 WebSocket 流式返回规划过程。
8. 前端实时展示生成内容。
9. 生成完成后保存最终结果。
10. 用户可查看历史记录并重新生成。

### 7.2 LLM 输出过程

流式阶段建议：

- 阶段 1：理解需求，确认起点、目的地、范围和交通方式。
- 阶段 2：查询天气，输出天气预警。
- 阶段 3：查询地图和 POI，生成路线候选。
- 阶段 4：按交通方式生成路径点规划或公共交通规划。
- 阶段 5：生成高德路线链接和路径图预览。
- 阶段 6：补充途径景点和停留建议。
- 阶段 7：补充实时信息检索，包括新闻/交通管制、攻略/避坑参考。
- 阶段 8：汇总为最终行程建议。

### 7.3 管理后台流程

1. 管理员登录后台。
2. 查看用户列表。
3. 查看生成记录列表。
4. 按用户、时间、状态、交通方式筛选记录。
5. 查看单条记录的输入、输出、错误信息和调用耗时。
6. 对异常记录执行重试、标记失败或删除。
7. 配置 LLM 供应商、API 地址、模型名和 API Key。
8. 测试 LLM 连接并启用/停用指定配置。

## 8. 数据实体清单

### 8.1 用户与权限

- users：用户。
- admin_users：管理员账号，或通过 users.role 简化实现。
- login_sessions：登录会话，按需。

### 8.2 生成记录

- generation_records：生成记录主表。
- generation_inputs：用户输入快照。
- generation_outputs：最终输出内容。
- generation_stream_events：流式输出事件，按需保存。
- generation_errors：异常信息，按需拆表。

### 8.3 外部数据快照

- route_snapshots：地图路径和 POI 查询快照。
- route_map_exports：高德路线链接、路径图截图或静态图资源。
- weather_snapshots：天气查询快照。
- news_snapshots：实时信息检索快照，按分类保存新闻/交通管制、攻略/避坑参考。
- llm_call_logs：大模型调用日志。

### 8.4 系统配置

- llm_configs：LLM 供应商、API 地址、模型名、API Key 密文、启用状态。
- config_audit_logs：配置变更记录，记录操作者、时间和变更摘要，不记录明文 Key。

详细 MySQL 表结构见 `docs/db/mysql_schema.md`。
