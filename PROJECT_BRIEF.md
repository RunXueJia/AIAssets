# 路书匠项目立项清单（简化版）

## 1. 项目命名

推荐项目名：路书匠

英文名：RouteCraft

一句话定位：

用户输入起点、目的地、范围和交通方式后，系统通过大模型流式生成一份可直接参考的出行规划，内容包含天气预警、路线/公共交通规划、途径景点和最近新闻。

备选名称：

- 行程铸造局
- 路线制图所
- 出发计划局
- 路书工厂
- 一路成行
- 行迹师
- 途编
- 云路书

建议使用“路书匠”，原因是它强调规划结果，不像复杂 SaaS，也不会显得过度 AI 化。

## 2. 项目目标

### 2.1 产品目标

- 做一个轻量的 AI 出行规划工具。
- 用户不需要复杂下单流程，只需要提交出行条件。
- 系统以流式方式展示大模型规划过程和最终结果。
- 后台只保留用户管理和生成记录管理。
- 第一版优先验证规划体验、生成速度和结果可用性。

### 2.2 用户输入

必填：

- 起点：城市、地址、景点或站点。
- 目的地：城市、地址、景点、区域或主题描述。
- 范围：可接受的距离、时间范围、游玩半径或天数。
- 交通方式：自驾、公共交通、步行/骑行、混合出行。

选填：

- 出行日期或大致时间。
- 人数。
- 偏好：自然风光、人文、亲子、美食、小众路线、低强度等。
- 避免项：不走高速、少换乘、少步行、避开热门景点等。

### 2.3 系统输出

通过 LLM 流式输出：

- 需求理解和规划假设。
- 天气预警：天气概况、恶劣天气提醒、出行风险。
- 路径点规划：适用于自驾、步行、骑行或混合路线。
- 公共交通规划：适用于地铁、公交、火车、高铁、打车接驳。
- 途径景点：推荐途经点、停留建议、顺路理由。
- 最近新闻：目的地或沿途相关的近期资讯、活动、交通管制提醒。
- 时间安排：按半日、全天或多日输出行程节奏。
- 风险提示：营业时间、限流、天气、交通、公共交通末班时间等。
- 可复制的结构化结果：Markdown 和 JSON 摘要。

## 3. 核心原则

- 先做简单可用，不做复杂订单、套餐、支付、人工审核。
- 用户端围绕“提交需求、看流式生成、查看历史记录”设计。
- 管理后台只做“用户管理、生成记录管理”。
- 大模型是核心体验，高德、天气、新闻等数据源作为辅助增强。
- 所有外部数据都标注来源和更新时间。
- 第一版不追求完整路书交付，只追求规划建议清晰、可读、可继续追问。

## 6. 角色设计

只保留两类角色：

- 游客/用户：提交出行规划需求、查看流式生成过程、查看生成结果、管理自己的历史记录。
- 管理员：管理用户、查看生成记录、查看调用状态、处理异常记录。

不做：

- 规划师角色。
- 运营角色。
- 复杂 RBAC 权限模型。
- 支付、订单、套餐、退款流程。

## 7. 核心流程

### 7.1 用户生成流程

1. 用户进入规划页。
2. 填写起点、目的地、范围、交通方式。
3. 按需填写日期、人数、偏好和避免项。
4. 点击生成。
5. 后端创建生成记录。
6. 后端聚合外部数据：地图、天气、新闻。
7. 后端调用 LLM，以 SSE 或 WebSocket 流式返回规划过程。
8. 前端实时展示生成内容。
9. 生成完成后保存最终结果。
10. 用户可查看历史记录并重新生成。

### 7.2 LLM 输出过程

流式阶段建议：

- 阶段 1：理解需求，确认起点、目的地、范围和交通方式。
- 阶段 2：查询天气，输出天气预警。
- 阶段 3：查询地图和 POI，生成路线候选。
- 阶段 4：按交通方式生成路径点规划或公共交通规划。
- 阶段 5：补充途径景点和停留建议。
- 阶段 6：补充最近新闻、活动、交通管制或风险提醒。
- 阶段 7：汇总为最终行程建议。

### 7.3 管理后台流程

1. 管理员登录后台。
2. 查看用户列表。
3. 查看生成记录列表。
4. 按用户、时间、状态、交通方式筛选记录。
5. 查看单条记录的输入、输出、错误信息和调用耗时。
6. 对异常记录执行重试、标记失败或删除。

## 14. 风险清单

- 公共交通实时性强，结果必须提示用户出发前复核。
- 天气、新闻和交通管制变化快，必须展示数据更新时间。
- AI 可能编造地点、时间或新闻，需要尽量以外部数据约束。
- 高德、天气、新闻接口可能失败，需要降级生成。
- 流式输出中断时要保存已有内容，避免用户完全丢失结果。
- 用户输入可能过于模糊，需要在 Prompt 中做假设说明。
- 生成记录可能包含个人位置数据，需要限制后台访问和日志外泄。
